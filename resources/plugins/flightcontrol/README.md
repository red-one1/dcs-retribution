# Moose FlightControl plugin (Retribution)

## Purpose

This plugin enables MOOSE `FLIGHTCONTROL` ATC at Retribution airbases.

It creates one `FLIGHTCONTROL` instance per eligible airbase and applies a consistent baseline configuration from Retribution plugin options, while allowing mission-specific customization through a hook.

## What it configures

For each configured airbase, the plugin:

- Creates `FLIGHTCONTROL:New(airbaseName, frequencyMHz, modulation)`
- Applies shared policy options:
  - landing/taxi limits
  - landing interval
  - taxi speed limit
  - runway repair time
  - hold pattern marker visibility
  - subtitles
  - radio transmission policy
   - optional SRS/TTS transmission suppression for testing (`disableSrsRadio`)
  - verbosity
- When `enableParkingGuards=true`, configures a default FLIGHTCONTROL parking guard as a one-unit ground group using `Soldier M4` for blue airfields and `Soldier AK` for red airfields
- When parking guards are enabled, hooks FLIGHTCONTROL `Birth` and `EngineStartup` handling to set AI flights ready for takeoff so taxi clearance can proceed and guards can be removed in mission setups where readiness is not set by mission tasking
- When parking guards are enabled, runs startup reconciliation passes that bind already-occupied parking aircraft to the local controller (when possible) and remove orphan guards for aircraft that are not controlled by that FLIGHTCONTROL instance
- Optionally disables built-in DCS ATC (`SetRadioSilentMode(true)`)
- Starts the controller (`fc:Start()`)
- Stores instances in `FlightControlRetributionControllers[airbaseName]`

## Airbase selection

Airbases are sourced from `dcsRetribution.Airbases` (generated by Retribution mission generation).

An airbase is configured only when:

- it has a valid `name`
- it has `side` equal to `"blue"` or `"red"`
- coalition enable switch is on (`enableBlue` / `enableRed`)

Neutral airbases are intentionally skipped.

## Frequency strategy

### Design goal

Avoid hardcoded airbase frequency tables in Retribution and reuse pydcs/DCS airport metadata when available.

### Mapping rule

If `useDcsAirbaseAtcFrequencies=true`, frequency source is selected per airbase in this order:

1. `atc_uhf_hz`
2. `atc_vhf_high_hz`
3. `atc_vhf_low_hz`
4. `atc_hf_hz`

The first available value is converted from Hz to MHz and used.

### Fallback behavior

If no ATC radio field exists (or if DCS-based selection is disabled), plugin falls back to:

- `blueFrequency` for blue airbases
- `redFrequency` for red airbases
- modulation from fallback option `useFM` (`FM` if true, else `AM`)

### Modulation decision

For DCS/pydcs-derived airbase frequencies, modulation is currently set to `AM`.

Rationale:

- Keeps behavior predictable across mixed UHF/VHF/HF picks
- Avoids ambiguity because pydcs ATC fields are provided as band frequencies, not explicit per-band modulation policy for this plugin
- Matches common ATC usage expectation for fixed-wing operations

## Integration points

- Plugin metadata/options: `plugin.json`
- Runtime setup script: `flightcontrol.lua`
- Data source population: `game/missiongenerator/luagenerator.py` exports per-airbase radio fields into `dcsRetribution.Airbases`

## Mission customization hook

You can define a global function before/alongside plugin execution:

```lua
function FlightControlRetributionConfigure(fc, airbaseName, side)
  -- Example: add holding patterns, custom SRS voices, parking guard policy, etc.
  -- fc:AddHoldingPattern("Batumi Hold Alpha", 320, 15, 6, 12, 10)
end
```

This hook is called once per created controller after common options are applied and before `Start()`.

## Key design decisions

1. **Per-airbase controller model**
   - Mirrors MOOSE `FLIGHTCONTROL` intent (`FLIGHTCONTROL:New` is airbase-scoped).

2. **Use generated mission data instead of static frequency tables**
   - Retribution already depends on pydcs and has airport ATC metadata.
   - Exporting metadata into mission Lua keeps runtime script simple and data-driven.

3. **Deterministic frequency selection with explicit fallback**
   - Mapping order (`UHF -> VHF high -> VHF low -> HF`) is stable and logged.
   - Coalition fallback ensures plugin remains robust when metadata is missing.

4. **Safe defaults, optional realism upgrades**
   - Plugin provides baseline behavior out of the box.
   - Hook enables advanced mission-specific FC usage without forking plugin core.

5. **Non-invasive with DCS ATC control option**
   - DCS ATC can be silenced to avoid duplicate/overlapping comms when desired.

## Notes

- Editor lint warnings about `env`, `radio`, `dcsRetribution` are expected in plain Lua static analysis; these globals are available in DCS mission runtime.
- If future missions require band-priority customization, add an option for preferred band order and resolve it before fallback.
- For crash/isolation tests, set plugin option `disableSrsRadio=true` to keep FLIGHTCONTROL active while suppressing its SRS/TTS transmissions.
